# 📢 全局黑话与绝对等号映射表 (Macro Dictionary)

- **黑话等式**：当遇到 **“mob1”**、**“mob1客群”** 或 **“mob1提取”** 时，**绝对等同于**索要完整的【首贷 + 完全结清 + 7天未复借流失】全套逻辑！
- **强制执行动作**：听到“mob1”，必须**毫无保留地**写全以下所有步骤：首贷精准锁定、完全结清判断、结清满7天前置校验、LEFT JOIN 的7天流失反连接剔除、埋点双重时间刹车。绝对不许只写“结清”而漏写“流失”！
- **工程强制约定**：只要遇到“流失客群提取埋点”、“抽样轨迹”等复杂需求，严禁图省事一镜到底！必须采用数仓标准工程：使用 CTE (WITH...AS) 分步建立临时表（圈人群 -> 找流失 -> 撞埋点），条理必须清晰。

# ⚙️ 全局通用业务逻辑与硬核约束 (Global)

- **新老客基础定义**：进件表中 customer_type = 'new' 为新客，customer_type = 'old' 为老客。（注意：精准首贷判定还需结合后续的 ROW_NUMBER 约束）。

- **真实借款判定**：必须产生具体的提现流水号（withdraw_uuid IS NOT NULL 或 withdraw_risk_uuid IS NOT NULL 且 != ''）才算一笔真实的借款订单。

- **分期订单“完全结清”判定（核心硬逻辑，强制必写）**：

1. 分期对齐：必须包含 HAVING COUNT(1) > 0 AND COUNT(1) = SUM(CASE WHEN asset_finish_at > asset_grant_at THEN 1 ELSE 0 END)。
2. **观察期防数据泄露（极重要！）**：必须在 SQL 中加入 CAST(MAX(asset_finish_at) AS DATETIME) <= DATE_SUB(CURRENT_DATE(), INTERVAL 7 DAY)，确保结清满 7 天才能进入流失判定，否则会引发严重右偏误判，少写扣分！

- **埋点抓取防 OOM 与双边界规则（强制必写）**：

1. **时间下界（严禁漏写）**：埋点时间必须 >= 结清时间。
2. **时间上界**：埋点时间必须 <= 结清时间 + 7天。

- **有序抽样约束**：任何抽取 N 个样本的需求（如 LIMIT 100），必须带上明确的排序条件（如 ORDER BY mob1_final_finish_time ASC），保证抽样结果绝对稳定可复现，严禁裸写 LIMIT！

### 1.巴基斯坦业务线 (Pakistan - PAK007 / CCP) 业务线特点：侧重于从注册到首贷全链路的漏斗流失分析。

客群 A：注册未进件

业务定义：完成了 App 账号创建，但没有发起预审（未提交基础资料）。

物理推导：dwd_w_user 存在记录，但关联 dwd_w_ask_loan_detail 时记录为空（is_apply=0）。

客群 B：进件审批通过未提现

业务定义：提交了资料且系统给出了预估额度（预审通过），但最终没有点击确认借款生成真实订单。

物理推导：dwd_w_ask_loan_detail.allow_loan = '1'（预审通过），但关联 dwd_w_apply.apply_status='1' 为空。

客群 D：MOB1 首贷结清流失

时间窗口与流失判定：业务采用固定 37 天窗口期。即第 0 天（正审放款日 asset_grant_at）往后推 37 天（30 天贷款周期 + 7 天观察期）。

物理推导：在这整整 37 天内，如果没有产生第二次下单时间（apply_create_at），则被判定为彻底流失。

### 2.墨西哥业务线 (Mexico - MEX017 / MexiCash) 业务线特点：多期借款产品的首贷流失与 eKYC 拦截分析。

MOB1 首贷结清流失客群

首贷锁定：通过按用户聚合，按订单申请时间升序排列取第一笔（ROW_NUMBER() = 1 且 loan_seq = 1）。

流失时间窗口：在首贷的所有分期完全结清后的 7 天内。

物理推导：以最后一期的实际结清时间 final_finish_at 为起点，往后推 7 天（INTERVAL 7 DAY）。若期间在 dwd_w_apply 中没有产生新的提现流水号（withdraw_uuid），即算流失。

eKYC 拦截客群（被拒挽回分析）

业务定义：在 eKYC 环节被系统明确拒绝的用户。

物理推导：底表 lty_mex017_decline 中 kyc_decline = 1。

观察窗口：业务设定观察用户在被拒后的 36 小时内（代码中为 36 * 60 * 60 * 1000 毫秒，虽然注释写的是 24 小时，实际执行的是 36 小时）是否有重试等埋点动作。

### 3.印尼业务线 (Indonesia - INA001 / Pinjamin / Pinjamwinwin) 业务线特点：底层数仓结构与 MEX 有差异，但业务流失判定逻辑高度对齐。

表结构与字段映射（印尼特有）：

用户表：不用 dwd_w_user，而是 dwb_c_user。

提现流水号：不用 withdraw_uuid，而是风控口径的 withdraw_risk_uuid。

时间字段：不用 user_create_time 和 withdraw_created_at，而是 user_create_at 和 withdraw_create_at。

MOB1 首贷结清流失客群

业务定义与时间窗口：完全同墨西哥（总期数 = 已结清期数，结清后 7 天无复借下单 withdraw_risk_uuid 动作即算流失）。

复借过滤严谨性：在匹配 7 天内的新单时，额外增加了一个自反逻辑 a.withdraw_risk_uuid != m.withdraw_risk_uuid，确保捕捉到的是真正的新复借单，排除了系统数据延迟导致查到首贷单自己的情况。

### 4.# 🛡️ 反欺诈特征挖掘 (Anti-Fraud Feature Engineering) - 全局 SOP

- **核心工程逻辑**：反欺诈特征开发不执行 SQL 数据提取，其标准模式是：**输入 (DataFrame: 用户特定窗口内的埋点数据) -> 处理 (Python 函数) -> 输出 (DataFrame: 特征列)**。

- **强制基础底表**：特征回溯必须使用 dwt.dwt_rsk_apply_info_base_d (申请)、dwt.dwt_rsk_withdraw_info_base_d (提现) 或 dwt.dwt_rsk_ask_loan_info_base_d (预审)。

- **回溯时间窗口**：默认抓取目标事件 (apply_create_at) 前 **14 天** 的埋点数据。

- **特征判定黑话**：

- 1. **Rapid Input (快速录入)**：银行卡录入总时长 <= 9秒。
  2. **Paste/Script (粘贴/脚本)**：银行卡输入步数 < 输入长度的 50%。
  3. **Blind Signing (盲签)**：用户未点击查看任何 term-btn (协议) 或 privacy-policy (隐私政策)。
  4. **Zero Slider (滑块零交互)**：在首页金额选择环节，normal-home-slide:change 事件触发次数为 0。

- **存储约定**：最终产出必须落入 dm_feature 库，且必须包含核心事件 ID (如 apply_uuid)。

# 📋 埋点字段在反欺诈中的核心语义

- eventname = 'bank-online-account-input:input': 其 extend 中的 value_len 代表输入长度。
- eventname = 'normal-home-slide:change': 代表滑块拖动行为。
- eventname = 'submit-btn:click': 结合 extend 中的 amount 代表用户最终申请意向。
