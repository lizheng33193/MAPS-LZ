我认真把这个仓库从入口、目录、主链路、模块分层、前端、扩展子系统都过了一遍。先给你一个最重要的结论：

这个项目本质上不是“一个神秘的大模型项目”，而是一套**面向墨西哥市场的用户画像分析平台 demo**。  
它的核心目标是：**给定一个用户 UID，汇总 App 安装、行为埋点、征信数据，生成多维画像、图表、报告，以及后续的产品/运营建议。**

而且还有个很关键的认知，能帮你先降复杂度：

**它名字里叫 multi-agent，但主干并不是一堆自由发挥的智能体乱跑。**  
主干其实是很工程化的后端流水线：`API -> Orchestrator -> 各画像模块 -> 汇总结果 -> 前端展示`。  
真正比较“agentic”的部分，主要是两个：
1. 自然语言对话编排器 `orchestrator_agent`
2. 独立的数据获取子系统 `data_acquisition_agent`

**这项目主要在做什么**

它做的是“用户画像 + 风险判断 + 客群分层 + 策略建议”。

你可以把它理解成 6 个业务层结果：

1. `App Profile`：看用户装了哪些 App，推断借贷风险、金融成熟度、消费能力  
2. `Behavior Profile`：看行为埋点，推断活跃度、还款意愿、流失风险、触达偏好  
3. `Credit Profile`：看征信/信用数据，推断债务压力、信用稳定性、借贷饥渴度  
4. `Comprehensive Profile`：把前面三类信号合并，做最终客群分层和总体判断  
5. `Product Advice`：基于综合分层，给产品策略  
6. `Ops Advice`：基于综合分层，给运营/催收/召回策略

**这个项目最核心的设计思想**

我觉得你最该抓住的是这 4 个点：

1. **规则先行，LLM 后置**  
   它不是让大模型直接“猜你是什么人”。  
   先由代码规则算出结果，再让 LLM 负责解释、润色、出报告。  
   这点在 App 模块特别明显。比如：
   [app/runtime_skills/app_profile_agent.py](<D:/WorkSpace/agent-user-profile-fork/app/runtime_skills/app_profile_agent.py>)
   [app/scripts/app_profile_payload_builder.py](<D:/WorkSpace/agent-user-profile-fork/app/scripts/app_profile_payload_builder.py>)

2. **模块分层很统一**  
   App、Behavior、Credit、Comprehensive 基本都被整理成同一套骨架。  
   所以你学会一个模块，其他模块会顺很多。

3. **数据输入是“按 uid 切好的本地文件”优先**  
   主运行时默认走 `LocalUserRepository`，也就是从 `data/app|behavior|credit/by_uid/` 下拿数据：  
   [app/repositories/local_repository.py](<D:/WorkSpace/agent-user-profile-fork/app/repositories/local_repository.py>)  
   这说明 demo 的运行模式是“先准备好数据，再跑画像”。

4. **这个仓库已经在往平台化演进**  
   不只是画像，还有：
   - 模块级渐进加载 `/api/analyze-module`
   - SSE 流式进度 `/api/analyze-stream`
   - Trace 深度行为解析 `/api/trace/{uid}`
   - 自然语言 Chat 编排器
   - 数据获取 Agent

- 多头借贷风险
- 金融成熟度
- 消费能力
- 活跃度和时间线
